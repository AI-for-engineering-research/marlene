1	# VIIRS Contrail Detection
2	
3	Experiment to evaluate whether an existing contrail detection model — trained on GOES-16 ABI data — can be applied to VIIRS data, either zero-shot or with light fine-tuning.
4	
5	## Motivation
6	
7	The ABI-trained model operates on geostationary imagery at 2 km spatial resolution at nadir. VIIRS (Visible Infrared Imaging Radiometer Suite) aboard Suomi-NPP/NOAA-20 offers higher spatial resolution (500 m for regular-grid product provided by NASA) and similar but not identical spectral channels. If the model generalizes to VIIRS with little or no retraining, it unlocks a fast gateway for contrail detections in higher-resolution satellite imager for multi-sensor observational contrail verification purposes.
8	
9	## Research Questions
10	
11	1. Does the ABI-trained model produce usable contrail detections on VIIRS imagery out of the box (zero-shot transfer)?
12	2. Which channel mapping between ABI and VIIRS bands works best as a drop-in replacement?
13	3. How much labeled VIIRS data (if any) is needed to close the performance gap through fine-tuning?
14	
15	## Approach
16	
17	- **Input data**: VIIRS Level-1B granules (bands mapped to approximate ABI equivalents)
18	- **Adapter**: `viirs_detection/viirs_adapter.py` loads native VIIRS bands, applies LUT conversion and bowtie fill, and exposes `detect_contrails_viirs()` — a self-contained wrapper around the mcast model that accepts explicit normalisation statistics and returns raw softmax probabilities (no threshold applied), enabling threshold sweeping without modifying mcast
19	- **Normalisation**: per-channel means and standard deviations computed from 48 stratified CONUS swaths (see Band Normalisation Data) and stored in `data/viirs_norm_stats_{c14sub}_for_C14.npz` — one file per C14 substitute (M15 and I5)
20	- **Detection model**: ABI-trained contrail segmentation model (`mcast`, version detection-1.1), loaded directly from weights
21	- **Evaluation**: probability maps are reprojected onto the label mask grid and thresholded; pixel-wise precision, recall, F1, and IoU are computed against binary ground-truth masks derived from manual annotations

133	## Bowtie Deletion Handling
134	
135	VIIRS M-band L1B data contains bowtie deletion zones — pixels where adjacent scan lines overlap at high viewing angles are removed in the gridded product. Analysis of scene 1 revealed:
136	
137	- **404 patches** of invalid pixels, all large (1648–3296 pixels each), zero isolated invalid pixels
138	- Each patch spans exactly **4 rows × ~1008 columns**, repeating at regular **~16-row intervals** (one per aggregation zone)
139	- Deletions are confined to the **left third (cols 0–1007) and right third (cols 2192–3199)** of the swath; the nadir-centred third has none
140	- ~13% of all pixels are invalid
141	
142	Unhandled, these NaN values propagate through the model's convolutional layers and suppress all detections across ~80% of the swath width.
143	
144	**Fix:** `fill_along_track` fills each deletion zone by column-wise linear interpolation along the along-track (row) direction, using the nearest valid pixels above and below each gap as control points. Since each gap is only 4 rows (~3 km) and the nearest valid pixels are at most 6 rows away, the interpolated values are a close physical approximation of what the sensor would have measured. NaN at the top or bottom of a column are filled with the nearest valid value; columns that are entirely NaN fall back to the band mean.

220	## Band Normalisation Data
221	
222	To compute VIIRS-derived normalisation statistics (per-channel means and standard deviations for M14, M15/I5, M16), swaths are randomly sampled from 2024–2025 CONUS overpasses using a stratified design with N samples per cell across four axes:
223	
224	| Axis | Values |
225	|---|---|
226	| Season | DJF, MAM, JJA, SON |
227	| Overpass | day, night |
228	| Location | west, central, east |
229	| Platform | NOAA-20, NOAA-21, Suomi NPP |
230	
231	4 seasons × 2 overpass types × 3 locations × 3 platforms × N = **72N swaths**.
232	
233	A global minimum 48-hour gap is enforced between any two selected swaths so the sample spans distinct meteorological conditions. With two years of data (~365 available 48h slots), N=3 is feasible; N=3 yielded 202 swaths (216 target — 14 cells are sparse night/west or night/east cells with too few candidates to fill all three slots).
234	
235	The pipeline runs on `hex` and consists of four tools (in a separate repository):
236	
237	- **`viirs_swath_finder`** — queries the VIIRS granule database (from node `g01` or `c018`), filters to CONUS overpasses, and assigns each granule to a (season, overpass, location) cell.
238	- **`viirs_swath_sampler`** — draws N=2 granules per cell reproducibly (fixed random seed), producing the final list of 48 swaths.
239	- **`viirs_batch_writer`** — writes a SLURM download batch script for each sampled swath. Takes `-fmr` (MOD radiances filename) and `-fmg` (MOD geolocation filename) as arguments.
240	- **`viirs_l1b_downloader`** — downloads each granule's MOD and IMG files (both radiances and geolocation, 4 files per granule) to `hex`. Both MOD and IMG filenames are resolved via LAADS directory listing (`browse_for_laads_filename`) rather than constructed from the parquet filename, so reprocessed granules with updated production timestamps are found correctly. All downloads and directory lookups have retry logic (up to 10 attempts, ~15 s wait with jitter) to handle transient LAADS traffic limits. The batch writer adds a SLURM `--error` file per job; deleting empty `.err` files after the run leaves only genuine failures for inspection.
241	
242	Statistics are then computed locally using `scripts/compute_norm_stats`:
243	
244	```bash
245	python scripts/compute_norm_stats [--l1b-dir /path/to/l1b_folder]
246	```
247	
248	This processes all `*02MOD*.nc` files in the folder in a single pass. For each swath it also looks for a matching `*02IMG*.nc` / `*03IMG*.nc` pair; if found, I5 statistics are accumulated on the native 375 m IMG grid (no resampling). Both sets of statistics are updated incrementally using a vectorised Welford algorithm so all swaths never need to be held in memory simultaneously. Two output files are written to `data/`:
249	
250	- `viirs_norm_stats_M15_for_C14.npz` — keys `M14_mean/std`, `M15_mean/std`, `M16_mean/std`, `BTD_M15_M16_mean/std`
251	- `viirs_norm_stats_I5_for_C14.npz` — keys `M14_mean/std`, `I5_mean/std`, `M16_mean/std`, `BTD_M15_M16_mean/std`
252	
253	If IMG files are absent for some swaths, those swaths are skipped for I5 with a warning.
254

        <div class="section-label">Spotlight · Summer 2026</div>
        <h1 class="section-title">Detecting contrails in VIIRS imagery</h1>
        <p class="section-sub">Testing whether an existing contrail detection model trained on GOES ABI imagery can be applied to VIIRS data with only little tuning.</p>
      </div>

      <!-- ─── INTRO ─────────────────────────────── -->
      <p class="sp-intro">
        The ABI-trained <code>mcast</code> model operates on geostationary imagery at 2 km resolution. VIIRS aboard Suomi-NPP and NOAA-20/21 offers 500 m resolution and similar — but not identical — spectral channels. If the model generalizes to VIIRS with little or no retraining, it unlocks higher-resolution contrail detections for multi-sensor observational verification.
      </p>

        <p class="sp-text">
          The pipeline takes raw VIIRS Level-1B granules, maps bands to their closest ABI equivalents, applies VIIRS-derived normalization statistics, and runs the unmodified <code>mcast</code> detection model. Two model versions are being evaluated (<code>detection-1.1</code> and <code>marvelous-moth-137</code>), and the output is a float32 probability map that can be swept over thresholds without re-running inference. Completed steps are shown in green; pending work in orange.
        </p>

        <p class="sp-text">

The M-band product (750 m) has no channel near 11 µm, so M15 (10.76 µm) is used as a stand-in for both C13 and C14. The I-band product (375 m) adds I5 at 11.45 µm — a much closer match to C14 at 11.2 µm — but requires downloading a second data product and reprojecting it onto the M-band grid.

      <!-- ─── BOWTIE & RESULTS ───────────────────── -->
      <div class="sp-section">
        <div class="sp-section-label">Zero-Shot Results</div>
        <h2 class="sp-section-title">Bowtie handling &amp; detection performance</h2>

        <p class="sp-text">
          VIIRS M-band L1B data contains bowtie deletion zones — pixels removed where adjacent scan lines overlap at high viewing angles. About 13% of pixels are invalid, concentrated in the left and right thirds of each swath in repeating 4-row gaps. Unhandled, NaN values propagate through the model's convolutional layers and suppress all detections across ~80% of the swath width.
        </p>
        <p class="sp-text">
          Three strategies were tested, each improving substantially on the last. The current fix — <em>along-track interpolation</em> — fills each 4-row gap by column-wise linear interpolation using the nearest valid pixels above and below, a physically reasonable approximation given the small spatial extent (~3 km) of each gap.
        </p>
